在 2 核 2G 的小机器上，跑起一个完整的个人站

架构 部署 SQLite

这是一个关于"如何在最小的机器上跑起一个完整可用个人站"的笔记。 没有花哨的微服务，没有 Redis 集群，没有 k8s——只有一个 2 核 2G 的 ECS， 一个 SQLite 文件，一个 PM2 守护的 Node 进程，和一个 nginx 反代。

为什么不上云数据库？

云数据库便宜的一年也要几百，而个人站的数据量一年可能不到 10MB。 用 SQLite 的好处显而易见：

  • 零运维：没有连接池，没有跨地域同步，没有主从切换
  • 零成本：本地文件，备份用 sqlite3 .backup 即可
  • 够用：单表百万级数据下，查询比 PG 还快（数据都在内存里）
  • 迁移简单：整个站点是 1 个文件 + 几张图片，scp 一下就到新机器

架构全景

整个架构画出来就是这样的：

[用户浏览器]
    ↓
[阿里云 CDN]  ← 缓存公开页面/图片/JS/CSS
    ↓
[nginx (反代 :80/:443)]
    ↓
[Nuxt 3 Node 进程 (PM2 守护, :3000)]
    ↓
[SQLite 文件 (/data/db.sqlite, WAL 模式)]
    ↓
[阿里云 OSS]  ← 图片/附件存储

注意：CDN 兜底了 90% 的静态请求，Node 进程只处理私密页面和后台管理。 所以即使 2 核 2G，也不会有压力。

内存预算：精打细算

2G 内存要装：系统 ~500MB、Node ~500MB、SQLite ~50MB、nginx ~50MB。 剩下的不到 1G 用来给文件系统缓存和突发流量。

关键配置：Node 启动参数 --max-old-space-size=512， PM2 配置 max_memory_restart=600M，超了就自动重启。

数据备份：一天一次不能少

SQLite 文件丢了一切就没了。我的备份策略：

  1. 每天凌晨 3 点 cron 跑 scripts/backup.sh
  2. 脚本用 sqlite3 .backup 做热备份（不锁表）
  3. 压缩后传到阿里云 OSS 的 backups/ 目录
  4. 保留 7 天滚动 + 月度归档
  5. 备份脚本带校验码，失败时发邮件告警

CDN 配置的几个坑

第一次配 CDN 的时候踩了不少坑。总结一下：

  • 回源 Host 必须用 ECS IP，不能用域名（避免回环）
  • 缓存策略要分层：HTML 5 分钟，JS/CSS 1 年，图片 30 天
  • 必须开启 HTTPS：阿里云免费 DV SSL 一年一续
  • 防盗链要设置：Referer 白名单 + URL 鉴权（防刷）

下一步

基础跑通之后，下一步要做的：

  • TipTap 编辑器 + Markdown 导入（P4 阶段）
